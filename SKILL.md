# 周待办提取

## 工作流总览

```
阶段一 前置确认（渠道选择→接入检测→配置引导→多组织→输出位置→通知）
阶段二 设定范围（时间范围→搜索对象）
阶段三 信息检索（逐个渠道执行→合并去重→排除待办记录文档）
阶段三B 完整性检查（群覆盖→分页→时间→关键词→跨渠道）
阶段四 待办提取（按6条标准筛选→分类排序→格式化）
阶段五 写入和发送（写入指定位置→可选通知责任人）
```

## 阶段一：前置条件确认和准备

### 1.0 配置持久化（静默加载）

执行本阶段前，先检测 `~/.config/wubc-extracting-weekly-todos/config.json` 是否存在。

**如果配置文件存在：**
  → **静默加载**，跳过 1.3~1.5
  → 除非用户主动说"帮我改一下XXX"，才更新对应项后重新保存

**如果配置文件不存在：**
  → 走 1.1~1.5 完整流程，完成后自动保存配置文件

**保存路径：** `~/.config/wubc-extracting-weekly-todos/config.json`

**保存内容：**
  - 钉钉多组织配置（org 列表 + corp_id）
  - 输出位置（feishu_doc / dingtalk_doc / ima_note）
  - 是否通知责任人（true / false）
  - 单聊联系人列表
  - 群聊名称列表
  - 更新时间戳

配置文件示例见 [config.example.json](config.example.json)


按以下顺序与用户对话确认配置。每步确认后再继续下一步。

### 1.1 渠道选择

列出全部可用渠道让用户选择：

```
可用渠道：
1. 钉钉 - 群聊、单聊
2. 飞书 - 云文档、妙记录音、单聊、群聊
3. IMA - 知识库、笔记
4. 印象笔记 - 笔记
```

确定用户选择了哪些渠道和子渠道，保存为 `selected_channels` 列表。

### 1.2 接入检测

对每个已选渠道执行检测。详细检测方法见 [references/02-channel-setup.md](references/02-channel-setup.md)。

| 渠道 | 检测方式 |
|------|---------|
| 钉钉 | 运行 `dws auth status` 或检测组织配置文件 |
| 飞书 | 运行 `lark-cli auth check` |
| IMA | 检测 `~/.config/ima/client_id` 和 `~/.config/ima/api_key` |
| 印象笔记 | 检测 `~/.codex/skills/yinxiang-skill/.env` 中的 token |

未接入的渠道：引导用户完成配置（具体引导步骤见 02-channel-setup.md）。

### 1.3 多组织确认（仅钉钉）

询问用户是否涉及多个钉钉组织。如果涉及，引导用户获取并保存各组织的 token（见 02-channel-setup.md 多组织部分）。

### 1.4 输出位置选择

询问用户周待办写到哪里：

```
可选输出位置：
1. 飞书云文档（需要 lark-cli + docx:write 权限）
2. 钉钉文档（需要 dws drive 权限）
3. IMA 笔记（需要 IMA API write 权限）
```

### 1.5 是否通知责任人

询问用户是否要将"团队完成"类待办通过钉钉私聊发送给各责任人。

## 阶段二：设定搜索范围（静默加载）

阶段二的配置遵从同一原则：**用户一经明确，不再重复询问**。

**如果 config.json 中有 search 字段（contacts 和 groups）：**
  → 静默加载，向用户展示一行信息提示（不打断、不询问）：
     `[当前搜索范围：联系人(张三、李四) | 群聊(全部) · 如需修改请直接说]`
  → 除非用户说"帮我加一个人"、"删掉 XXX 群"等，才更新并重新保存

**如果 config.json 中没有 search 字段：**
  → 首次使用，按以下步骤让用户填写

### 2.1 时间范围

默认**近7天**（从当天往前推7天）。可选自定义天数或起止日期。时间范围不持久化（每次按当前日期计算近7天即可）。

### 2.2 搜索对象

- **单聊联系人**：询问用户需要提取哪些联系人的私聊消息。如果用户不知道，可用 `dws chat list --private` 或 `lark-cli im search --private` 列出近7天有消息的联系人供选择。
- **群聊名称**：可选，用户可指定群名或留空（留空=搜索全部有消息的群）。
- 填写完成后，**自动保存到 config.json 的 search 字段**。

## 阶段三：信息检索

按用户选择的渠道顺序，逐个执行信息检索。详细命令和 API 见 [references/03-search-methods.md](references/03-search-methods.md)。

### 检索原则

1. **按时间发现，不按关键词过滤**：先列出近7天全部有更新的群/文档/消息，再逐个读取内容，确保不遗漏。
2. **不重复提取待办记录文档**：明显是"待办事项记录类"的文档不作为信息源（见 [references/06-duplicate-prevention.md](references/06-duplicate-prevention.md)）。

### 合并去重

所有渠道的记录按时间正序排列。同一事项出现在多个渠道时，保留最早来源并标注"多渠道来源"。

## 阶段三B：检索完整性检查

信息检索完成后、待办提取之前，执行完整性检查。逐项验证后向用户报告结果。检查清单见 [references/03-search-methods.md](references/03-search-methods.md#完整性检查) 中的完整性检查部分。

检查内容：群发现覆盖率、分页完整性、时间范围、关键词覆盖（飞书文档搜索）、跨渠道一致性。

检查失败时自动补搜：漏群则重新 list 并读取，分页未读完则继续翻页，关键词不足则补充搜索词。

## 阶段四：提取待办事项

### 待办判定标准

一条记录同时满足以下**全部 6 条**才列为待办：

| # | 条件 | 说明 |
|---|------|------|
| 1 | 来源合规 | 来自用户选择的渠道，在近7天范围内 |
| 2 | 未完成 | 截至提取日，该事项未在近7天内闭环完成 |
| 3 | 具体工作 | 指向具体工作或任务，非咨询/询问类 |
| 4 | 责任明确 | 由用户自己完成，或由用户亲自布置给团队成员 |
| 5 | 含动作 | 含明确或隐含的动作要求（做、确认、推进、回复等） |
| 6 | 时间合适 | 应在下周完成（含由近7天顺延的） |

### 排除清单

以下情况**不列为待办**：

- **已完成**：对话闭环（"已提交"、"👌"后无追问）
- **咨询/询问**：一句话能决定的非持续性任务（"这个发哪里？"、"怎么设置？"）
- **非我安排**：总部通知、同事间互相@的任务（不是你布置或指向的）
- **信息同步**：纯信息告知无动作要求（"今天推文已发布"）
- **待办记录文档**：工作计划、周报等文档内容本身（防循环提取）

### 分类和排序

仅两类，每类按紧急程度排序：

```
类别一：由我完成
  有明确截止日期的 → 紧急标记🔥 → 排前
  上级指令 → 排前
  无截止日期 → 排后

类别二：团队完成（我布置的）
  有明确截止日期 → 排前
  跟上级汇报相关 → 排前
  无截止日期 → 排后
```

### 输出格式

每条待办必须包含：内容 + 来源 + 责任人 + 截止时间。

详细格式规范见 [references/05-output-format.md](references/05-output-format.md)。

## 阶段五：写入和发送

### 5.1 写入指定位置

将整理好的待办事项写入用户选择的输出位置（飞书云文档/钉钉文档/IMA 笔记）。

写入格式：Markdown，按分类和紧急程度排版。默认追加（保留历史），用户也可要求覆盖。

### 5.2 通知责任人

如果用户选择了"通知责任人"，对"团队完成"分类的每条待办：
1. 提取责任人姓名
2. 通过 `dws chat send` 发送钉钉私聊
3. 同一责任人多条待办合并为一条消息发送

通知模板见 [references/05-output-format.md](references/05-output-format.md)。

## 关键原则

1. **渠道选择由用户动态决定**，非硬编码
2. **待办记录类文档不作为信息源**，避免循环提取
3. **完整性检查是必须环节**，每个渠道检索后执行
4. **责任人为团队其他人时，必须是由用户布置的**，非我安排不列入
5. **通用化设计**：面向企业平台所有同事，须引导用户完成前置配置
