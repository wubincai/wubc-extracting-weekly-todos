## 钉钉群聊

步骤：
1. 列出近7天活跃群：`dws chat list --group --time <7天前时间戳>`
2. 如果用户指定了群名称，用名称匹配过滤
3. 读取每个群的消息：`dws chat list --group --name "<群名>" --time <时间范围>`
4. 检查分页：如果返回结果中有 hasMore=true，继续翻页读取

多个组织时：对每个组织分别执行上述步骤。

## 钉钉单聊

步骤：
1. 对每个指定的联系人，先用私聊列表查找：`dws chat list --private --time <7天前时间戳>`
2. 找到联系人后，读取私聊消息：`dws chat list --private --userid <userid> --time <时间范围>`
3. 检查分页

如果用户不知道 userid：先用 `dws chat list --private --time` 列出近7天所有私聊，让用户辨认。

## 飞书云文档

步骤：
1. 用关键词搜索：`lark-cli doc search --keyword "<关键词>" --time <时间范围>`
2. 建议使用多组搜索词确保覆盖：`7月`、`2026`、月份名称
3. 读取搜索结果中每篇文档的内容
4. 检查分页：hasMore / page token
5. 检查每篇文档的 updated 字段是否在近7天内

注意：飞书文档搜索依赖关键词，单一关键词可能漏掉标题不匹配的文档。

## 飞书妙记录音

步骤：
1. 列出妙记：`lark-cli minutes list --time <7天前时间戳>`
2. 对每条妙记，提取关键词摘要
3. 检查每条妙记的 start_time 是否在近7天内

注意：飞书妙记只有关键词摘要，没有全文。

## 飞书单聊 & 群聊

### 飞书单聊
1. 列出近7天私聊：`lark-cli im search --private --time <7天前时间戳>`
2. 对每个指定联系人读取消息
3. 检查分页

### 飞书群聊
1. 列出近7天活跃群：`lark-cli im search --group --time <7天前时间戳>`
2. 读取消息：`lark-cli im read --chat <chat_id>`
3. 检查分页

## IMA 知识库

步骤：
1. 列出所有知识库：IMA API `GET /knowledge-bases`
2. 对每个知识库列出文档：`GET /knowledge-bases/{id}/documents`
3. 读取文档内容：`GET /knowledge-bases/{id}/documents/{doc_id}`

注意：IMA 知识库 API 不返回条目的创建/修改时间戳，标注⚠️。

## IMA 笔记

步骤：
1. 列出笔记：IMA API `GET /notes`
2. 筛选更新时间在近7天内的笔记
3. 读取笔记内容
4. 检查分页

## 印象笔记

步骤：
1. 获取笔记列表：印象笔记 API `GET /notes`（带时间过滤参数）
2. 筛选更新时间在近7天内的笔记
3. 读取笔记内容
4. 检查分页（hasMore）

## 完整性检查

信息检索完成后、待办提取之前，执行以下逐项检查：

### 钉钉-群聊
| 检查项 | 方法 | 判定 |
|--------|------|------|
| 群发现覆盖率 | 发现的群数 = 已读取数 | 数量一致 |
| 时间范围 | 消息最早最晚时间戳 | 覆盖近7天 |
| 分页完整性 | 每个群 hasMore=false | 全部读完 |

### 钉钉-单聊
| 检查项 | 方法 | 判定 |
|--------|------|------|
| 联系人覆盖率 | 用户指定数 = 已读数 | 全读到 |
| 时间范围 | 消息时间跨度 | 覆盖近7天 |
| 分页完整性 | hasMore=false | 全部读完 |
| 多组织去重 | 同一联系人出现在多个组织时选数据更全的组织 | 不重复 |

### 飞书-云文档
| 检查项 | 方法 | 判定 |
|--------|------|------|
| 搜索结果完整性 | 返回总数 vs 实际读取数 | 无截断 |
| 关键词覆盖 | 至少使用2组搜索词（月份+年份） | 覆盖到位 |
| 时间范围 | 每篇文档的 updated 字段 | 全部在近7天 |
| 分页 | hasMore/page token | 无遗漏 |

### 飞书-妙记
| 检查项 | 方法 | 判定 |
|--------|------|------|
| 时间范围 | 妙记 start_time | 全部在近7天 |
| 内容摘要 | 关键词是否提取完整 | 检查通过 |

### IMA 知识库
| 检查项 | 方法 | 判定 |
|--------|------|------|
| 知识库全覆盖 | 全部知识库已搜索 | 数量一致 |
| 时间过滤 | 不支持精确过滤时标注 | 建议人工确认 |

### IMA 笔记 & 印象笔记
| 检查项 | 方法 | 判定 |
|--------|------|------|
| 时间范围 | 笔记修改时间 | 全部在近7天 |
| 分页 | hasMore | 无遗漏 |

### 跨渠道一致性
| 检查项 | 方法 | 判定 |
|--------|------|------|
| 同一事项多渠道出现 | 合并去重时检查 | 标注"多渠道来源" |
| 时间连续性 | 消息时间是否有断档 | 如有标记 |

### 检查结果输出格式

简要报告格式：
```
🔍 检索完整性检查结果

✅ 钉钉-群聊：N个群完整读取，无遗漏
✅ 飞书-云文档：N篇全部读取
⚠️ IMA 知识库：无法按时间精确过滤，建议人工确认

共计读取XX条记录。
```

### 检查失败处理
| 失败类型 | 处理方式 |
|---------|---------|
| 群遗漏 | 重新 list 后读取遗漏的群 |
| 分页未读完 | 继续翻页到 hasMore=false |
| 关键词不足 | 补充搜索词后重搜 |
| IMA 不可精确过滤 | 标注，建议用户手动确认 |
| 跨渠道矛盾 | 以消息源发时间为准 |

### 检查关键原则
1. API 能力决定检查精度
2. 宁可标记也不要遗漏数据
3. 检查报告可追溯
4. 多组织钉钉：每个组织单独检查
5. 最终兜底：检查后询问用户是否有遗漏
