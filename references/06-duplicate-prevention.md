## 为什么需要防循环

如果飞书"7月工作计划"文档本身就是一份待办记录文档，把它当作"信息来源"去提取待办，会形成：
```
提取得到了"完成7月工作计划"这条待办
然后把这个待办写入"7月工作计划"文档
下周读取该文档时又把它当作来源提取出来
无限循环
```

## 判断标准

符合以下**任一特征**的文档/笔记，**不作为信息源提取**：

### 标题匹配
文档标题包含以下关键词：
- `待办`、`工作计划`、`周报`、`任务清单`
- `Gantt`、`进度表`、`OKR`、`KPI`
- `月报`、`季报`、`年度计划`、`规划`

### 内容结构匹配
文档内容以表格为主，且表格列标题包含：
- `事项`、`任务`、`工作项`
- `完成时间`、`截止`、`DDL`
- `责任人`、`负责人`、`执行人`
- `状态`、`进度`、`完成情况`

### 用途判断
文档是用户自己创建的**事件记录类文档**，如：
- "季度规划"、"月度总结"、"2026年Q3计划"
- "个人OKR"、"团队KPI"、"冲刺清单"

## 例外规则

如果用户**明确指定**某一待办记录类文档作为参考源（如"看看我上个月的待办完成情况"），则以用户指令优先。

## 跨渠道去重

同一事项出现在多个渠道时的处理规则：

| 场景 | 处理方式 |
|------|---------|
| 同一件事在钉钉和飞书都有讨论 | 保留最早来源，标注"多渠道来源" |
| A 渠道的信息是 B 渠道的补充 | 合并为一条，标注多渠道 |
| 两个渠道内容完全一致 | 只保留一条 |

去重后结构化保持：
```json
{
  "content": "下周五前完成营销活动方案",
  "source": "多渠道来源",
  "source_details": [
    "钉钉-XXX群-XXX 7/3 14:22",
    "飞书-群聊-XXX群-XXX 7/3 15:01"
  ],
  "responsible": "自己",
  "deadline": "7/10"
}
```
