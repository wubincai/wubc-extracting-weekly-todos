# 待办事项提取规则

## 判断标准

从全部数据源的原始工作记录中，逐条判断是否应列入下周待办：

| # | 规则 | 说明 | 示例 |
|---|------|------|------|
| 归类 | 规则 | 判断条件 | 示例 |
|------|------|---------|------|
| 🔴 **硬性截止** | ① 下周有明确截止 | 有"下周一前""下周X前"等具体截止时间 | "审计整改下周一下班前交" |
| 🔴 **硬性截止** | ② 本周截止未确认 | 截止日已过或在本周内，但无完成确认标记 | "周四前完成模块梳理"→周四已过未确认 |
| 🟡 **未完成/待推进** | ③ 提到"下周"待行动 | "从下周开始""下周再对接"等，确定需要行动 | "海宁试点从下周开始" |
| 🟡 **未完成/待推进** | ④ 有行动项未完成 | "下一步""需对接""需推进""需解决"等措辞 | "会员中心需对接互联网中心排期" |
| 🟡 **未完成/待推进** | ⑤ 未解决问题/异常 | 待处理的异常、故障、风险、bug | "淘宝闪购C转P异常需补充方案" |
| 🟡 **未完成/待推进** | ⑥ 分配/负责的事项 | 用户直接安排或他人分配给用户的任务 | "AI团队组建你亲自负责" |
| 🟢 **周期性工作** | ⑦ 固定频率必做 | 按周/月固定频率要做的常规事项 | 周分析、周报、月报 ||

## 排除规则

| 排除类型 | 说明 | 示例 |
|---------|------|------|
| 已完成 | 事项已有明确完成确认标记 | 已完成的活动/已结束的项目 |
| 纯通知 | 仅信息传达，无行动项 | 总部下发的活动通知（无用户行动项） |
| 被明确停止 | 有"暂缓""不做了""暂停"等 | 已决定暂停的项目 |

## 冲突处理原则

同一事项在多个来源中都被提及时：

1. 合并为一条，保留最完整的信息
2. 来源列写全部来源渠道，用 ` + ` 连接（如 `飞书云文档 + 飞书妙记`）
3. 责任人取最具体的信息

## 责任人判断规则

| 情况 | 处理 |
|------|------|
| 来源中明确指定了责任人 | 直接使用 |
| 来源中提到"你负责""你亲自""你安排" | 判断为用户本人 |
| 来源中提到部门/团队 | 写为"团队" |
| 无法判断 | 留空 |


- 同一事项在不同数据源中出现 → 合并为一条
- 保留最完整的信息（截止时间、责任人）
- 取覆盖信息最全的版本
- 来源列合并显示（如 `飞书妙记 + 飞书云文档`）
