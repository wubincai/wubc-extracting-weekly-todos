# 待办事项提取规则

## 判断标准

从全部数据源的原始工作记录中，逐条判断是否应列入下周待办：

| # | 规则 | 说明 | 示例 |
|---|------|------|------|
| ① | **明确下周截止时间** | 有"下周一前""下周二前""下周X前"等字眼 | "审计整改下周一下班前交" |
| ② | **提到"下周"** | "从下周开始""下周来商量" | "海宁试点从下周开始" |
| ③ | **有截止日但未确认完成** | 本周有 deadline 但未确认完成状态 | "周四前完成模块梳理"→周四已过未确认 |
| ④ | **周期性工作** | 每周/每月固定要做的事项 | 周分析、周报、月报 |
| ⑤ | **有行动项但无完成标记** | "下一步""需对接""需推进""需解决" | "会员中心需对接互联网中心排期" |
| ⑥ | **未解决问题** | 待处理的异常、问题、bug | "淘宝闪购C转P异常需补充方案" |
| ⑦ | **亲自负责或分配的任务** | 用户直接安排的工作、分配的任务 | "AI团队组建你亲自负责" |

## 排除规则

| 排除类型 | 说明 | 示例 |
|---------|------|------|
| 已完成 | 事项已有明确完成确认标记 | 已完成的活动/已结束的项目 |
| 纯通知 | 仅信息传达，无行动项 | 总部下发的活动通知（无用户行动项） |
| 被明确停止 | 有"暂缓""不做了""暂停"等 | 已决定暂停的项目 |

## 冲突处理原则

同一事项在多个来源中都被提及时：

1. 合并为一条，保留最完整的信息
2. 来源列写全部来源渠道，用 ` + ` 连接（如 `飞书云文档 + 飞书妙记`）
3. 责任人取最具体的信息

## 责任人判断规则

| 情况 | 处理 |
|------|------|
| 来源中明确指定了责任人 | 直接使用 |
| 来源中提到"你负责""你亲自""你安排" | 判断为用户本人 |
| 来源中提到部门/团队 | 写为"团队" |
| 无法判断 | 留空 |

## 分类规则

提取出的待办事项按性质归入五个分类：

| 分类 | 适用条件 |
|------|---------|
| 🔴 **硬性截止** | 有明确的截止日期/时间，且截止日在下一周 |
| 🟡 **未完成/待推进** | 有行动项但未完成、需跟进推进的事项 |
| 🟠 **部门级重点工作** | 部门级的重要项目/方向性工作，持续推动 |
| 🟢 **周期性工作** | 按周/月固定频率要做的常规工作 |
| 🔵 **持续推进** | 长期项目中的持续性动作，无明确截止日 |

## 去重原则

- 同一事项在不同数据源中出现 → 合并为一条
- 保留最完整的信息（截止时间、责任人）
- 取覆盖信息最全的版本
- 来源列合并显示（如 `飞书妙记 + 飞书云文档`）
